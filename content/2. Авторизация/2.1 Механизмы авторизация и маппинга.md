
### BR-20-000 Бесшовная авторизация в продуктах Atlassian

**Ревизия:** 04.2026

Пользователь (руководитель) хочет войти в систему, используя свою рабочую учетную запись Atlassian, и одним действием предоставить сервису доступ ко всей своей экосистеме (Jira, Confluence, Bitbucket Cloud). Смысл для пользователя — получить доступ сразу ко всем своим рабочим пространствам без необходимости повторной авторизации для каждого отдельного сайта. После входа система должна автоматически определить все доступные пользователю ресурсы, а пользователь — выбрать, какие из них активировать для мониторинга в дашборде.

### BR-210-001 Формирование единой структуры проекта (Mapping)

**Ревизия:** 04.2026

Руководитель проекта нуждается в инструменте, который позволит связать данные из разных источников в единую сущность «Проект». Смысл в том, чтобы ИИ-ассистент понимал технический контекст: какие репозитории (из Bitbucket Cloud или внешних GitHub/GitLab) и какие страницы документации относятся к конкретному проекту в Jira. Пользователь должен иметь возможность гибко настраивать эти связи, чтобы аналитика по коммитам и документам корректно сопоставлялась с задачами сотрудников.

## FEAT-20-000 Интеграция с экосистемой Atlassian (OAuth 2.0)

**Макеты:**
### SR-20-000-000

**Теги:** #priority/high 

**Ревизия:** 01.04.2026

Система должна перенаправлять пользователя на сервер авторизации Atlassian (https://auth.atlassian.com/authorize) при нажатии кнопки «Login with Atlassian». Запрос должен содержать:
- client_id: Из настроек приложения в Developer Console.
- scope: offline_access, read:jira-user, read:jira-work, read:confluence-content.summary, read:repository:bitbucket (или аналогичные для доступа к Bitbucket Cloud).
- prompt=consent: Для принудительного отображения экрана выбора сайта и согласия с правами.
- audience=api.atlassian.com.
- state: Уникальный токен для защиты от CSRF.

### SR-20-000-001

**Теги:** #priority/high 

**Ревизия:** 01.04.2026

При получении authorization_code на эндпоинт /auth/callback, бэкенд обязан обменять его на пару access_token и refresh_token через POST-запрос к https://auth.atlassian.com/oauth/token. Полученный access_token должен обеспечивать доступ ко всем сайтам (cloud_id) пользователя, на которые были выданы разрешения.

### SR-210-000-002

**Теги:** #priority/high 

**Ревизия:** 01.04.2026

Поскольку приложение находится в статусе **PRIVATE**, система должна корректно обрабатывать ошибку доступа для пользователей, не включенных в список Contributors. В случае ошибки «This application is in development», фронтенд должен отобразить экран-заглушку с объяснением, что доступ на этапе разработки ограничен, и предложением обратиться к администратору для добавления email в список разрешенных.

## FEAT-210-001 Управление рабочими пространствами и маппинг ресурсов

**Макеты:**
### SR-210-001-000

**Теги:** #priority/high 

**Ревизия:** 01.04.2026

После получения access_token, система должна выполнить запрос к https://api.atlassian.com/oauth/token/accessible-resources для получения списка всех доступных пользователю рабочих пространств (cloud_id). Все найденные инстансы Jira должны быть отображены пользователю в интерфейсе настроек с указанием их имен, URL и аватаров.

### SR-210-001-001

**Теги:** #priority/high  

**Ревизия:** 01.04.2026

Для каждого выбранного пользователем рабочего пространства система должна создавать отдельную запись в БД. Каждая запись должна содержать индивидуальный cloud_id сайта, но **идентичные** значения access_token и refresh_token, принадлежащие данному пользователю. Это обеспечит корректное отображение списка пространств и возможность работы с API для каждого из них.

### SR-210-001-002

**Теги:** #priority/high 

**Ревизия:** 01.04.2026

Если при попытке доступа к API конкретного рабочего пространства возвращается ошибка 403 (недостаточно прав, пользователь не является администратором сайта), система должна визуально выделить данное пространство и вывести статус «Требуется подтверждение администратора». Интерфейс должен содержать ссылку на инструкцию о том, как предоставить приложению "Alp Agent OAuth" необходимые права через настройки Atlassian.

### SR-210-001-003

**Теги:** #priority/high 

**Ревизия:** 01.04.2026

Интерфейс должен позволять подключать внешние Git-сервисы (GitHub/GitLab) через отдельный OAuth-флоу. Система должна предоставлять функционал «Маппинга», где пользователь вручную связывает Jira-проекты с репозиториями (как из Bitbucket Cloud, доступного автоматически, так и из внешних подключенных сервисов). Для внешних сервисов связывание происходит путем сопоставления URL репозитория с ключом проекта в Jira.

### SR-210-001-004

**Теги:** #priority/high

**Ревизия:** 01.04.2026

Бэкенд должен проверять валидность access_token перед каждым запросом к API (или за 5 минут до истечения expires_at). В случае истечения, система должна выполнить запрос на обновление через refresh_token. После получения новой пары токенов, бэкенд обязан **синхронно обновить все записи** пользователя в таблице токенов для всех его рабочих пространств.
