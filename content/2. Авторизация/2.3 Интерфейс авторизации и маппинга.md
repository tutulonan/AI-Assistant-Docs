
### BR-230-000 Безопасный вход через Atlassian ID

**Ревизия:** 04.2026

Руководитель хочет иметь единую точку входа в сервис через Atlassian OAuth 2.0. Смысл для пользователя — отсутствие необходимости ручной регистрации: при первом входе система автоматически создает учетную запись на основе данных профиля Atlassian, а при повторных входах исключает дублирование аккаунтов. Пользователь должен четко понимать, к каким данным (Jira, Confluence, Bitbucket Cloud) он дает доступ, зная, что один полученный токен обеспечит работу со всеми его рабочими пространствами (сайтами) одновременно.

### BR-230-001 Выборочный мониторинг и настройка ресурсов

**Ревизия:** 04.2026

После авторизации Руководитель хочет видеть все свои доступные площадки и иметь возможность выбрать те из них, которые станут частью единой структуры проекта. Смысл для пользователя — система должна сама определить, к каким сервисам (Jira, Git, Confluence) есть доступ, и автоматически предложить логические соответствия между ними (например, связать проект и репозиторий с похожими именами). Пользователь должен иметь возможность гибко настроить эти связи, чтобы в дальнейшем аналитика строилась только на основе релевантных данных.

### BR-230-002 Объединение кода и задач в единый проект

**Ревизия:** 04.2026

Руководитель часто использует внешние репозитории (GitHub/GitLab) в связке с Jira. Смысл для пользователя — возможность «привязать» эти внешние источники к проектам Jira внутри нашего сервиса для формирования сквозной аналитики. Система должна применять фильтрацию на основе выбранных связей, чтобы руководитель видел коммиты, задачи и документацию как единый поток в рамках одного проекта, даже если системы не связаны на стороне провайдеров.

### SR-230-000-001

**Теги:** #priority/mid 

**Ревизия:** 01.04.2026

Рядом с кнопкой входа должен быть размещен краткий текст, поясняющий, что авторизация дает сервису право на чтение данных из Jira (задачи), Confluence (документация) и Bitbucket Cloud (код). Текст должен акцентировать внимание на том, что доступ ко всем сайтам пользователя предоставляется через единую авторизацию.

### SR-230-000-002

**Теги:** #priority/high 

**Ревизия:** 01.04.2026

Система должна обрабатывать негативные сценарии авторизации: при отмене действия пользователем выводить сообщение «Авторизация отменена», при технических сбоях или недействительном коде — «Не удалось авторизоваться. Повторите позже». При статусе приложения Private необходимо выводить текст о доступе только для Contributors.

## FEAT-230-001 Управление рабочими пространствами Atlassian

**Макеты:**
### SR-230-001-000

**Теги:** #priority/high  

**Ревизия:** 01.04.2026

После обмена кодов на токены система должна получить список всех доступных сайтов пользователя через `/accessible-resources` и информацию о профиле пользователя через `/me`. В БД должна создаваться отдельная запись на каждый сайт (cloud_id) с сохранением идентичных токенов доступа, при этом система должна блокировать создание дубликатов учетных записей для одного Atlassian ID.

### SR-230-001-001

**Теги:** #priority/high

**Ревизия:** 01.04.2026

Система должна автоматически определять набор доступных сервисов (Jira, Bitbucket, Confluence). Если список проектов или сайтов пуст, система обязана отобразить сообщение «Нет доступных данных для настройки». Пользователь выбирает один или несколько проектов Jira с помощью переключателей (Toggle) для активации сбора метрик.

### SR-230-001-002

**Теги:** #priority/high

**Ревизия:** 01.04.2026

Система должна блокировать сохранение настроек, если пользователь не выбрал ни одного элемента. В этом случае отображается сообщение: «Выберите хотя бы один проект, репозиторий или пространство». Если при активации мониторинга возвращается ошибка 403, должна отображаться кнопка «Инструкция по доступу» для связи с администратором сайта Atlassian.

### SR-230-002-000

**Теги:** #priority/mid

**Ревизия:** 01.04.2026

В интерфейсе должен быть выделен блок «Внешние источники кода» (GitHub/GitLab). После успешной авторизации через эти сервисы система должна обновить общий список доступных репозиториев для маппинга.

### SR-230-002-001

**Теги:** #priority/high

**Ревизия:** 01.04.2026

Система должна реализовывать интерфейс сопоставления данных, где на основе доступной информации автоматически предлагаются соответствия (например, Jira Project X <-> Git Repo X). **Теги:** #priority/high

**Ревизия:** 01.04.2026
