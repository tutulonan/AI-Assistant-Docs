### Git API: 

##### 1. Объект: Коммиты (Activity Monitoring)

Endpoint: GET /repos/{owner}/{repo}/commits

- (string) Уникальный ID. Ключ для связи с другими сущностями.
- (timestamp) Точное время активности.
- (string) ID пользователя для маппинга с Jira/Calendar. ==привязывают ли задачи к гиту?==
- (text) Парсинг через RegEx (\[A-Z]+-\d+) для поиска ключа задачи Jira.
- (int) Объем изменений.
- (string) Пути к файлам.

##### 2. Объект: Pull / Merge Requests (Cycle Time & Bottlenecks)

Endpoint: GET /repos/{owner}/{repo}/pulls

- (int) Ссылка на PR.
- (string) open, closed, merged.
- (string) Автор PR.
- created_at, updated_at, closed_at, merged_at: (timestamps).
	-Бизнес-логика: merged_at минус created_at. Среднее время поставки фичи.
- requested_reviewers\[]: (array) Список логинов, на ком висит задача.
	-Бизнес-логика: Список блокирующих лиц ("Кто тормозит процесс?").
- (string) Ветки "откуда" и "куда".
- (boolean) Наличие конфликтов с целевой веткой.
	-Бизнес-логика: Индикатор риска "Merge Hell".

##### 3. Объект: Ревью и Комментарии (Quality & Mentorship)

Endpoint: GET /repos/{owner}/{repo}/pulls/{pull_number}/reviews

- state: (string) APPROVED, CHANGES_REQUESTED, COMMENTED.
- submitted_at: (timestamp) Время реакции.
	-Бизнес-логика (Review Velocity): submitted_at (review) минус created_at (PR). Как быстро команда реагирует на чужой код.
- body: (text) Текст замечаний.
	-Бизнес-логика: Подается на вход ИИ для анализа токсичности или частоты однотипных ошибок (например, "пропустил тесты").

##### 4. Объект: Проверки и CI/CD (Stability)

Endpoint: GET /repos/{owner}/{repo}/commits/{ref}/check-runs

- status: (string) queued, in_progress, completed.
- conclusion: (string) success, failure, timed_out.
	-Бизнес-логика: Если conclusion == failure в основной ветке — критическое уведомление ИИ-боту "Прод упал/сборка сломана".

##### 5. Объект: Сравнение веток (Release Risks)

Endpoint: GET /repos/{owner}/{repo}/compare/{base}...{head}

- ahead_by: (int) Количество коммитов в ветке разработки, которых нет в master.
- behind_by: (int) Насколько master ушел вперед.
	-Бизнес-логика: Оценка сложности будущего деплоя. Если behind_by > 50, вероятность конфликтов при мердже близка к 100%.

##### 6. Объект: Анализ файлов (Ownership)

Endpoint: GET /repos/{owner}/{repo}/stats/contributors

Основные поля для БД:

- weeks[]: Массив недель с полями a (additions), d (deletions), c (commits).
	-Бизнес-логика: Построение тренда производительности команды. Спад c при росте a может значить, что разработчик закопался в сложном рефакторинге.


#### Что можно оценивать по данным с гита:

##### 1. Активность и нагрузка (Commits & Stats)

- Автор и Дата/Время: Позволяет строить графики активности. Метрика: Пики и спады производительности команды по дням/часам.
- Связь с задачей (Commit Message + Jira ID): Главный маркер прозрачности. Аналитика: Какой процент кода пишется без привязки к задачам (стихийная разработка).
- Объем изменений (Lines added/deleted): Не для оценки «строк в час», а для выявления аномалий. Аналитика: Слишком крупные коммиты (риск при ревью) или подозрительно высокая активность в непрофильных модулях.

##### 2. Эффективность процесса (Pull/Merge Requests)

- Time to Merge (Lead Time): Время от создания PR до его вливания в master. Метрика: Скорость поставки фич (TTM).
- Статус Ревью (Approved / Changes Requested): Кто из команды проводит ревью, а кто — нет. Аналитика: Выявление «бутылочных горлышек» (когда все ждут одного эксперта) и вовлеченность команды в проверку кода.
- Количество итераций (Comments / Updates): Сколько раз автор исправлял код после замечаний. Аналитика: Качество кода на входе (если итераций слишком много — задача была плохо проработана или разработчик не справляется).
- Статус CI/CD в PR: Проходят ли автотесты и сборки перед мерджем. Метрика: Стабильность кода и частота «сломанных» веток.

##### 3. Здоровье проекта и блокировки (Branches & Conflicts)

- Divergence (Ahead/Behind): Насколько ветки разработки отстали от основной. Аналитика: Риск «ада слияния» (Merge Hell). Чем дольше ветка живет отдельно, тем дороже её вливать.
- Stale PRs (Забытые задачи): Список PR, в которых нет активности более N дней. Метрика: Потерянный фокус и «зависшие» в работе задачи.
- Mergeable State (Конфликты): Наличие конфликтов в открытых PR. Аналитика: Прямой блокировщик релиза, требующий вмешательства.

##### 4. Релизы и поставка (Tags & Releases)

- Частота релизов (Release Frequency): Как часто тегируются новые версии. Метрика: Ритмичность поставки ценности бизнесу.
- Состав релиза (Release Notes): Автоматический список всех задач (из Commit Messages), вошедших в конкретный тег. Аналитика: Прозрачный отчет для бизнеса «что именно уехало на прод».

##### 5. Дисциплина (Process Compliance)

- Commits to Protected Branches: Случаи пуша напрямую в master/main в обход PR. Аналитика: Нарушение регламентов безопасности и качества.
- Авторство кода (Blame/Contributions): Распределение экспертизы по модулям. Аналитика: "Bus Factor" — если 90% критического модуля написано одним человеком, это риск для компании.
